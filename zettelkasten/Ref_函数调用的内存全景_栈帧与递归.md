# 函数调用的内存全景：栈、栈帧、递归与尾递归

> [!abstract] 核心观点
> 函数调用的本质是栈帧的压入与弹出。理解了栈帧，就理解了递归为什么会爆栈、尾递归为什么能优化、EVM 为什么限制调用深度为 1024。

---

## 1. 进程内存布局

> 程序运行时，内存被分成哪几块？

```
高地址
┌──────────────────┐
│     Kernel       │  操作系统内核空间（用户态不可访问）
├──────────────────┤
│   Stack  ↓       │  函数调用栈（向低地址增长）
│                  │
│   （空闲区间）    │
│                  │
│   Heap   ↑       │  动态内存（向高地址增长）
├──────────────────┤
│   BSS            │  未初始化全局/静态变量
├──────────────────┤
│   Data           │  已初始化全局/静态变量
├──────────────────┤
│   Text           │  程序代码（只读）
└──────────────────┘
低地址
```

进程的虚拟地址空间分为五个区域。**Text 段**存放可执行指令，只读，防止代码被意外覆写。**Data 段**存放已初始化的全局变量（如 `int x = 5;`），**BSS 段**存放未初始化的全局变量（运行时自动清零）。**Heap（堆）**是动态内存区域，由程序员通过 `malloc/new` 申请，从低地址向高地址增长，需要手动管理（或 GC）。**Stack（栈）**则从高地址向低地址增长，由编译器自动管理。

栈和堆的增长方向相反，它们从两端向中间"夹逼"。如果两者相遇，就会发生内存溢出（`stack overflow` 或堆溢出）。Linux 默认每个线程的栈大小为 **8 MB**（可通过 `ulimit -s` 查看），这也是递归深度受限的根本原因。

> [!tip] 类比
> 栈像一摞盘子——后放的先拿，顺序严格（LIFO）。堆像仓库——可以在任意位置存取货物，但需要记录位置（指针）和空间管理（防止碎片化）。

## 2. 栈与栈帧

> 调用一个函数时，栈上发生了什么？

```
高地址
┌─────────────────────┐
│   参数 b            │  ← 调用者 push（或寄存器传递）
├─────────────────────┤
│   参数 a            │
├─────────────────────┤
│   返回地址          │  ← call 指令自动 push（返回时跳回的位置）
├─────────────────────┤
│   保存的 RBP        │  ← 被调用者 push（保存调用者的帧基址）
├─────────────────────┤  ← RBP 指向此处（当前帧基址）
│   局部变量 result   │
├─────────────────────┤  ← RSP 指向此处（栈顶）
低地址
        int add(int a, int b) 的栈帧
```

每次函数调用，CPU 会在栈上分配一块专属区域——**栈帧（Stack Frame）**，也叫活动记录（Activation Record）。一个栈帧包含四类数据：

| 组件 | 内容 | 负责方 |
|------|------|--------|
| 参数 | 调用函数时传入的实参 | 调用者（Caller）|
| 返回地址 | 函数返回后 CPU 应跳回的指令地址 | `call` 指令自动压入 |
| 保存的帧指针 | 调用者的 RBP 值（用于恢复上层帧） | 被调用者（Callee）|
| 局部变量 | 函数内部声明的变量 | 被调用者 |

两个关键寄存器负责维护栈帧边界：**RSP（Stack Pointer）**始终指向当前栈顶（最低地址），**RBP（Base Pointer）**指向当前栈帧的底部（作为固定参考点，访问局部变量和参数都相对它偏移）。

**调用过程（3步）：** ① 调用者 push 参数 → ② `call` 指令自动 push 返回地址并跳转到函数入口 → ③ 被调用者 push 保存旧 RBP、设置新 RBP = RSP、sub RSP 分配局部变量空间。

**返回过程（3步）：** ① 将返回值放入 RAX 寄存器 → ② `mov rsp, rbp` 释放局部变量（弹出整个帧） → ③ `pop rbp` 恢复调用者帧指针，`ret` 弹出返回地址并跳回。

> [!tip] 动手验证
> 用 `gcc -S -O0 add.c` 生成汇编，可以清晰看到：`push rbp` / `mov rbp, rsp` / `sub rsp, 16` / `mov DWORD PTR [rbp-4], ...` / `leave` / `ret` 的完整序列。

## 3. 函数调用链

> 多层调用时，栈帧如何叠加和回收？

```
高地址
┌──────────────────────────┐
│  main() 栈帧             │ ← 最先压入，最后弹出
│  locals: argc, argv...   │
├──────────────────────────┤ ← main 调用 foo 时，RSP 下移
│  foo() 栈帧              │
│  locals: x, y...         │
├──────────────────────────┤ ← foo 调用 bar 时，RSP 再下移
│  bar() 栈帧              │ ← 当前执行位置，RSP 在此
│  locals: tmp...          │
├──────────────────────────┤ ← RSP（栈顶）
低地址

  bar 返回 → 弹出 bar 帧 → RSP 回到 foo 帧顶
  foo 返回 → 弹出 foo 帧 → RSP 回到 main 帧顶
```

多层调用时，栈帧按 **LIFO（后进先出）** 顺序叠加。每调用一层，RSP 向低地址移动，新帧压在旧帧之上；每次返回，RSP 向高地址移动，旧帧自动被释放（内存并不清零，只是"不再被使用"）。

这个机制的关键性质：**栈帧的生命周期与函数执行周期严格一致**——调用时自动分配，返回时自动释放，不需要程序员手动管理。这也是为什么 C/C++ 的局部变量不需要 `free`。

然而这也带来一个陷阱：**函数返回后，其栈帧所在的内存立即"失效"**。如果在函数内部返回了一个局部变量的指针，调用方拿到的是一个悬空指针（dangling pointer）——该内存可能随时被下一次函数调用覆写，是 C/C++ 经典 bug 来源之一。

## 4. 递归

> 递归为什么会爆栈？

```c
int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);  // 注意：乘法在递归返回之后执行
}
```

以 `factorial(4)` 为例，观察栈的增长过程：

```
调用 factorial(4)：

步骤1: 压入 f(4) 帧
┌──────────┐
│  f(4)    │ ← 等待 f(3) 返回后再做 4 * ?
└──────────┘

步骤2: 压入 f(3) 帧
┌──────────┐
│  f(3)    │ ← 等待 f(2) 返回后再做 3 * ?
├──────────┤
│  f(4)    │
└──────────┘

步骤3: 压入 f(2) 帧
┌──────────┐
│  f(2)    │ ← 等待 f(1) 返回后再做 2 * ?
├──────────┤
│  f(3)    │
├──────────┤
│  f(4)    │
└──────────┘

步骤4: 压入 f(1) 帧，触底返回 1
┌──────────┐
│  f(1)=1  │ ← 返回 1，弹出
├──────────┤
│  f(2)    │ ← 得到 1，计算 2*1=2，弹出
├──────────┤
│  f(3)    │ ← 得到 2，计算 3*2=6，弹出
├──────────┤
│  f(4)    │ ← 得到 6，计算 4*6=24，弹出
└──────────┘
```

关键洞察：`return n * factorial(n-1)` 中，**乘法是一个"待完成操作"**——f(4) 必须等待 f(3) 的结果才能完成自己的计算，因此 f(4) 的栈帧不能被提前销毁，必须在栈上保留。这意味着递归深度 = n = 同时存活的栈帧数。

当 n 足够大（例如 `n = 100000`），累积的栈帧会把 8MB 的栈空间耗尽，触发 **Stack Overflow**（段错误 / SIGSEGV）。

> [!tip] 调试技巧
> 用 gdb 在 `factorial` 上设置断点，每次中断时执行 `bt`（backtrace），可以看到调用栈随递归深度逐层增长的全过程。

## 5. 尾递归与优化

> 为什么尾递归可以不增长栈？

**普通递归（有待完成操作）：**
```c
int factorial(int n) {
    if (n <= 1) return 1;
    return n * factorial(n - 1);  // 返回前还要做乘法 → 帧不能释放
}
```

**尾递归（用累加器消除待完成操作）：**
```c
int factorial_tail(int n, int acc) {
    if (n <= 1) return acc;
    return factorial_tail(n - 1, n * acc);  // 最后一步是纯函数调用 → 帧可复用
}
// 调用：factorial_tail(4, 1)
```

**尾位置（Tail Position）的定义**：函数中某个调用表达式处于尾位置，当且仅当该调用之后没有任何其他操作——函数的返回值直接就是该调用的返回值。

| 特征 | 普通递归 | 尾递归 |
|------|----------|--------|
| 最后一个动作 | 乘法（二元运算） | 递归调用本身 |
| 调用返回后还有工作？ | 是 | 否 |
| 当前帧能否复用？ | 否（必须保留等待结果） | 是（可直接覆写） |
| 栈帧数量 | O(n) | O(1)（TCO 后）|

```
普通递归（4帧叠加）：     尾递归（1帧循环复用）：

┌──────────┐             ┌──────────────────┐
│  f(1)    │             │  f(n-1, acc)     │
├──────────┤             │                  │
│  f(2)    │    ──TCO──▶ │  同一个帧        │
├──────────┤             │  参数被覆写      │
│  f(3)    │             │  n-1, n*acc      │
├──────────┤             │  → 反复执行 →    │
│  f(4)    │             └──────────────────┘
└──────────┘                   ↑（原地复用，不增长）
```

编译器实现尾调用优化（TCO）的机制：将 `call`（压入新帧）替换为 `jmp`（直接跳转到函数入口，复用当前帧）。因为当前帧的返回地址和上层帧信息不再需要，覆写参数后直接跳转即可。

**注意事项（语言差异）：**
- gcc / clang：`-O2` 开启时自动执行 TCO
- Rust：不保证 TCO，推荐手动改写为迭代或使用蹦床（trampoline）模式
- Python：明确**不支持** TCO（设计者 Guido 认为保留完整调用栈对调试更重要）
- JavaScript：ES6 规范虽要求 PTC（Proper Tail Calls），但主流引擎 V8/SpiderMonkey 均未实现，仅 Safari/JSC 支持

尾递归的本质是一种**思维转换**：将"返回后还要做的事"提前计算好，打包成参数传递给下一次调用，从而消除"欠账"——帧不再是"等待结果的占位符"，而是可以随时丢弃的临时工作区。

## 6. 与 EVM 的对比

> EVM 的"栈"和原生栈有什么不同？

| 维度 | 原生调用栈（x86-64） | EVM 调用栈 |
|------|---------------------|------------|
| 深度限制 | 受栈内存大小约束（~10K+ 帧） | **硬限制 1024 层**（EIP-150 的 63/64 gas 规则使实际可用深度显著低于此） |
| 帧内容 | 参数、返回地址、局部变量、寄存器 | 调用上下文（gas、caller、calldata、returndata 等） |
| 数据存储 | 寄存器 + 栈内存 | Memory（线性字节数组）+ Storage（持久化 K-V） |
| TCO 支持 | 是（编译器优化） | **否**（每次 CALL 都创建新帧，无法复用） |
| 溢出后果 | 段错误（进程崩溃） | 交易 revert（状态回滚，gas 不退） |
| 典型溢出场景 | 无限递归 | 递归深度等于数组长度的算法 |

EVM 实际上有**两套独立的"栈"**：
1. **调用栈（Call Stack）**：记录 `CALL` / `DELEGATECALL` 的调用上下文，深度上限 1024。
2. **操作数栈（Operand Stack）**：EVM 的字节码执行引擎是栈式虚拟机，每条字节码指令操作的 256-bit 字值栈，深度上限 1024 个字。

两者概念不同，通常说的"1024 调用深度限制"指的是前者。

**案例分析：OZ #6289 quicksort 递归深度问题**

OpenZeppelin 的一个 quicksort 实现在最坏情况下（数组已近似有序时，每次 pivot 选取不均匀），递归深度等于数组长度。当数组长度约 170 时，调用深度接近 1024，触发 EVM revert。

**修复思路**：快速排序的调用深度由 *递归分区的层数* 决定，而非数组长度。选择"只对较小分区递归，较大分区用循环处理"的策略（Sedgewick 优化），可将最大递归深度从 O(n) 降至 O(log n)——对 170 个元素，约 8 层，远低于 1024 的限制。
