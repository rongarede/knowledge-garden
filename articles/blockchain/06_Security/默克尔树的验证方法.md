# Merkle Tree 白名单验证机制

使用 Merkle Tree 实现链上白名单校验机制，仅需保留根节点 (`merkleRoot`) 即可实现高效、低成本的成员资格验证。

## WHAT｜什么是 Merkle Tree 白名单？

一种链上成员验证方案，通过链下构建哈希树，仅存储 `root`，用户通过 `leaf + proof` 即可证明资格。OpenZeppelin 提供了 `MerkleProof.verify()` 用于链上验证。

## WHY｜为什么使用它？

- 减少链上存储和 Gas 消耗（只保存一个 `bytes32`）
- 保密性强，不暴露完整地址列表
- 适合大规模地址验证
- 可拓展性强，可将额度、阶段等信息加入 leaf

## WHO｜参与者

- 部署者 / 项目方：生成 Merkle Root，部署到合约中
- 用户 (EOA)：通过前端获得 proof 并在合约中调用验证函数
- 智能合约：持有 `merkleRoot`，负责校验 proof 并处理业务逻辑

## HOW｜如何实现？

链下流程（构建 Merkle Tree）：
- 收集地址列表 → `keccak256(addr.toLowerCase())` → 构建 Merkle Tree（`sortPairs: true`） → 输出 `root` 和每个地址的 `proof`

链上流程（Solidity 校验）：

```solidity
bytes32 leaf = keccak256(abi.encodePacked(msg.sender));
require(MerkleProof.verify(proof, merkleRoot, leaf), "Invalid proof");
```

校验通过则视为白名单用户，可继续执行 claim/mint 操作。

常见安全措施：
- 添加 `mapping(address => bool) claimed` 防止重复领取
- 保证链下 `leaf` 构造方式与链上完全一致
- 使用地址小写统一（如 `.toLowerCase()`）避免大小写不一致校验失败

## WHEN｜什么时候用？

- 白名单规模较大（>几十地址），不适合用 `mapping` 全部存储时
- 静态或分批可更新名单场景（如空投、NFT铸造白名单等）
- 不适用于名单频繁动态变更（如每秒变动的投票资格），可考虑 EIP-712 签名或 zk 方案

## 注意事项 / 踩坑警告

- 确保生成 Merkle Tree 时 `sortPairs` 与链上逻辑一致
- 多字段 `leaf` 时需保持编码顺序和类型一致（`abi.encodePacked()`）
- 链上添加防重复领取机制，例如 claimed 标志位或位图压缩
