# 代理合约（Proxy Contract）

代理合约通过 `delegatecall` 将逻辑与状态分离，实现"逻辑可升级、状态可延续"的架构目标。

## What（是什么）

- Proxy 负责保存状态并接收用户交互。
- Implementation 承载业务逻辑。
- 代理将调用转发给实现合约，升级时替换实现地址即可。

## Why（为什么使用）

- Solidity 合约部署后代码不可直接修改。
- Proxy 允许业务逻辑从 V1 迭代到 V2，同时保留历史状态。
- 可模块化拆分职责并复用实现合约。

## How（怎么实现）

核心机制：`delegatecall`

```solidity
(bool success, bytes memory data) = implementation.delegatecall(msg.data);
```

实现步骤：

1. 编写逻辑合约（如 `LogicV1`）。
2. 编写代理合约，在 `fallback()` 中转发调用。
3. 使用 `initialize()` 完成初始化。
4. 设计受控升级入口（如 `upgradeTo(address)`）。

## How much（成本/安全）

- `delegatecall` 一般比普通调用更复杂，需关注 gas 与可读性。
- 多代理复用实现合约可节省部署成本。

注意：升级时必须保证存储布局兼容；不兼容会直接破坏状态。

升级函数必须有权限控制；`constructor` 和 `immutable` 在代理上下文中行为与直连部署不同。

## 典型实现模式

| 模式 | 特点 |
|---|---|
| Transparent Proxy | 区分 admin/user，管理路径清晰 |
| UUPS Proxy | 升级逻辑在实现合约，更轻量 |
| Beacon Proxy | 用 Beacon 统一管理多个 Proxy |
