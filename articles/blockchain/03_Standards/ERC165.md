# ERC-165 标准介绍（4W1H）

## What（它是什么）

- ERC-165 是以太坊的一个**接口识别标准**，定义了智能合约如何声明自己支持哪些接口。
- 合约实现该标准后，任何其他合约或客户端都可以通过调用 `supportsInterface(bytes4 interfaceId)` 方法来查询它是否实现了某个接口。
- 示例：ERC721 合约可通过 `supportsInterface(0x80ac58cd)` 表明自己支持 NFT 标准。
- 定义于 EIP-165，提出于 2018 年，成为所有可组合智能合约的基础元协议。

## Why（为什么需要）

- Solidity 本身不支持运行时接口反射（如 Java 的 `instanceof`），在链上无法直接判断一个合约是否支持某个功能。
- ERC-165 解决了**智能合约在链上自描述接口能力的问题**。
- 其设计目标是：使不同合约、钱包、DApp、交易市场可以**自动化识别和兼容特定接口**。
- 提高了合约的可组合性、可拓展性，尤其在 NFT（ERC721）、多资产（ERC1155）中至关重要。

## When（提出与使用场景）

- 提案时间：2018 年 1 月
- 标准编号：EIP-165
- 创作者：Christian Reitwiessner（Solidity 核心开发者）

### 核心用途场景

- 检查是否为 ERC721 合约
- 判断合约是否支持某个扩展（如 Metadata、Enumerable）
- 合约可动态响应是否支持某类标准或插件接口
- 前端钱包/交易所/DApp 自动识别功能支持

## Where（在哪些系统中使用）

- ERC721 / ERC1155 合约中均使用 ERC165 注册自身接口
- OpenZeppelin 合约库中的 `ERC165` 模块作为基类
- OpenSea、MetaMask、Etherscan 等前端使用 `supportsInterface()` 来识别 NFT 或合约功能
- DAO 模块系统中，用于判断模块是否支持某个控制接口（如 `voting`, `staking`, `metadata`）
- Plug-in 合约结构设计（如 ERC2535 Diamonds）中用于注册不同接口能力

## How（如何实现）

- 每个接口有唯一的 `interfaceId`，为接口中所有函数选择器的 XOR 值（bytes4）
- 合约继承 `ERC165` 并实现 `supportsInterface(bytes4)` 方法

### 示例接口 ID 计算

```solidity
// IERC721 interfaceId = bytes4(keccak256("balanceOf(address)"))
//                  ^ bytes4(keccak256("ownerOf(uint256)"))
//                  ^ ...
```

### 示例 Solidity 实现方式

```solidity
contract MyContract is ERC165 {
    function supportsInterface(bytes4 interfaceId) public view virtual override returns (bool) {
        return interfaceId == type(IMyInterface).interfaceId || super.supportsInterface(interfaceId);
    }
}
```

- OpenZeppelin 提供 `ERC165`, `IERC165`, `supportsInterface()` 等安全封装模块

### 结合建议

- 建议所有可组合性合约实现 ERC165
- 建议配合 ERC1820 注册接口（更通用的注册中心）
